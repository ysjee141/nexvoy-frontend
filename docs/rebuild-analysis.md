# OnVoy 서비스 재개발 분석 및 제안

> 작성일: 2026-06-11  
> 현재 버전: Next.js 16 + Supabase + Capacitor 8  
> 서비스 상태: **정식 오픈 전 (운영 사용자 없음)**

---

## 1. 현황 진단

### 1.1 현재 아키텍처 요약

```
웹/앱 통합 단일 코드베이스
┌─────────────────────────────────────┐
│  Next.js (SSR/Static Export 토글)   │
│  ├── 20개 페이지 (App Router)       │
│  ├── 9개 API Routes                 │
│  ├── 13개 Service 클래스            │
│  └── 40개 컴포넌트                  │
├─────────────────────────────────────┤
│  Capacitor 8.2.0 (하이브리드 브릿지)│
│  → 정적 내보내기(out/) 래핑          │
└─────────────────────────────────────┘
         ↕ Supabase (PostgreSQL + Auth + Storage + RLS)
```

**빌드 전략의 핵심 구조:**
- 웹 빌드: `next build` → SSR 활성화
- 모바일 빌드: `BUILD_TARGET=mobile next build` → `output: 'export'` (정적)
  - API Routes를 임시 폴더로 이동 후 빌드, 완료 후 복원하는 스크립트 방식

### 1.2 사용자가 제기한 문제

| 문제 | 분석 |
|------|------|
| 여행 일정 페이지 진입 시 0.5~1s 딜레이 | Client Component이므로 hydration + Supabase 쿼리 대기 시간 |
| 사용자 눈에 보이는 로딩 | skeleton shimmer 처리 중이나 초기 fetch 시간 자체를 줄이기 어려움 |
| 앱 특화 기능 관리 분산 | 플랫폼 감지 코드(`Capacitor.isNative()`)가 10곳 이상에 흩어짐 |

## 2. 핵심 병목 원인 분석

### 2.1 로딩 딜레이의 실제 원인

현재 여행 일정 페이지(`/trips/detail`)가 느린 이유는 SSR 부재가 아니라 아래 구조적 한계 때문이다.

```
사용자 진입
  → HTML 수신 (빠름)
  → JS 번들 hydration (~200-300ms)
  → Supabase 인증 확인 (50-100ms)
  → trips 쿼리 (100-300ms, 네트워크 왕복)
  → plans 쿼리 (100-300ms)
  → 화면 표시
```

**총 체감 딜레이: 500ms~1s** — 이는 구조 문제이며, Next.js SSR을 웹에서 풀로 쓰더라도 인증된 사용자의 개인 데이터는 서버에서 미리 조회하기 위해 세션 쿠키 기반 서버 렌더링이 필요하다.

### 2.2 현재 통합 구조의 숨겨진 비용

| 항목 | 내용 |
|------|------|
| API Routes 복잡도 | 모바일 빌드 시 `prebuild`/`postbuild` 스크립트로 임시 이동 처리 |
| Sentry 제약 | 정적 내보내기에서 Sentry Tunnel Route 불가 → 모바일 오류 추적 한계 |
| 이미지 최적화 | 모바일 빌드에서 `unoptimized: true` 강제 → LCP 성능 저하 가능성 |
| Server Components | 정적 내보내기와 혼용 불가 → 전체 Client Component로 구성 필요 |
| Capacitor 커플링 | 13개 Service 중 7개가 Capacitor 의존 → 웹 전용 테스트 어려움 |

## 3. 선택지 비교

### Option A: 현상 유지 + 점진적 개선

현재 구조를 유지하면서 병목 지점만 선별 최적화한다.

**개선 가능 항목:**
- 여행 목록 페이지: `stale-while-revalidate` 캐싱 적용
- React Query / SWR 도입으로 클라이언트 캐시 레이어 추가
- 중요 페이지 prefetch 강화

**장점:**
- 추가 투자 없이 개선 가능
- 현재 1인 개발 체계 유지
- 리스크 최소

**단점:**
- 근본적 딜레이 해소 불가 (아키텍처 한계)
- 기술 부채 누적 (플랫폼 감지 분산, API Routes 이동 스크립트 등)
- 앱 성능 한계 유지 (WebView 기반)

---

### Option B: 웹 SSR 강화 (Capacitor 유지)

모바일은 Capacitor를 유지하되, 웹 빌드에서 Server Components와 SSR을 적극 활용한다.

**변경 범위:**
- 인증된 사용자 데이터를 Server Component에서 미리 fetch
- API Routes 이동 스크립트 제거 (웹 전용 API는 SSR에서 직접 처리)
- 모바일 빌드는 현재 방식 유지 (정적 내보내기)

**실질적 개선 효과:**
- 웹에서 초기 로딩 0.5~1s → 100~200ms로 단축 가능
- Sentry Tunnel 정상 동작 (웹)
- Image Optimization 활용 (웹)

**단점:**
- 웹과 앱 코드 경로가 달라져 복잡도 증가
- Server Component + Client Component 혼용 설계 필요
- Capacitor 앱은 여전히 WebView 기반 한계 유지

**추정 공수:** 3~4주 (페이지별 SSR 전환)

#### Option B의 복잡도 리스크

Option B는 "얼마나 깊이 들어가느냐"에 따라 성격이 크게 달라진다. 깊이 도입할수록 아래 문제가 현실화된다.

**1. Server/Client Component 경계 분열**

웹 SSR을 강화할수록 컴포넌트를 Server/Client로 나누게 된다. 그런데 모바일 빌드는 정적 내보내기라 Server Component를 쓸 수 없다. 결국 같은 페이지에 두 경로가 생긴다.

```
trips/detail/
  ├── page.tsx        ← 웹: Server Component로 prefetch
  └── page.client.tsx ← 앱: Client Component (현재 방식)
```

이 패턴이 20개 페이지에 퍼지면 어느 버전이 정답인지 파악이 어려워진다.

**2. "이 코드 웹에서만 되는 거였나?" 문제**

수개월 뒤 수정할 때 해당 코드가 웹 전용인지, 앱에도 적용되는지 즉시 파악이 어렵다. 주석과 규칙으로 보완할 수 있지만, 시간이 지날수록 실수가 생긴다.

**3. API Routes 이동 스크립트의 복잡도 가중**

SSR 강화 과정에서 API Routes가 늘어날수록 빌드 스크립트의 이동 대상도 늘어난다. 이 스크립트가 실패하면 모바일 빌드가 깨지는데, 디버깅이 어렵다.

**4. 테스트 신뢰도 저하**

SSR로 전환된 페이지는 웹 E2E가 통과해도 Capacitor 앱에서 다르게 동작할 수 있다. 두 경로를 모두 검증하려면 테스트 비용이 두 배가 된다.

**복잡도 통제 기준:**

| 적용 범위 | 효과 | 복잡도 증가 | 권장 여부 |
|-----------|------|------------|---------|
| React Query 캐시만 추가 | 딜레이 30~40% 개선 | 거의 없음 | 권장 |
| 목록 페이지 SSR 1~2개 제한 | 딜레이 50~70% 개선 | 관리 가능 | 조건부 권장 |
| SSR 전면 적용 (20개 페이지) | 최대 개선 | 높음 | 비권장 |

> SSR을 전면 도입하면 Option C와 비슷한 복잡도를 갖되 앱 성능 개선은 없는 최악의 조합이 될 수 있다.
> Option B를 선택한다면 **React Query 캐시 + 목록 페이지 SSR 1~2개** 선에서 멈추는 것이 복잡도 리스크를 통제하는 현실적 방법이다.

---

### Option C: 웹(Next.js SSR) + 앱(React Native) 완전 분리

두 개의 독립적인 프로젝트로 분리한다.

```
nexvoy-web (Next.js)          nexvoy-app (React Native)
├── 20개 페이지 (SSR)         ├── 20개 화면 (네이티브)
├── 9개 API Routes           │  (직접 Supabase SDK 호출)
└── Supabase 서버 렌더링       └── Expo 또는 bare RN
         ↕                              ↕
              Supabase (공유 DB)
```

**장점:**
- 웹: SSR/ISR/Server Components 풀 활용 → 최대 성능
- 앱: 네이티브 렌더링 → 스크롤, 애니메이션, 메모리 압도적 개선
- 코드 역할 명확 분리 → 유지보수성 향상
- 각 플랫폼 생태계를 제약 없이 활용 가능

**단점:**

| 항목 | 내용 |
|------|------|
| 초기 투자 | RN 프로젝트 세팅 + 20개 화면 재구성 → 예상 3개월 |
| 코드 중복 | 비즈니스 로직 상당 부분 두 곳에서 관리 필요 |
| 배포 파이프라인 | 웹(Vercel) + 앱(스토어) 별도 관리 |
| 테스트 중복 | E2E(Playwright) + 앱 테스트(Detox) 별도 관리 |
| 1인 개발 한계 | 두 프로젝트를 동시에 고품질로 유지하기 어려움 |

---

### Option D: 앱 전용 (React Native Only, 웹 폐기)

웹을 완전히 버리고 앱만 운영한다. React Native로 재작성하여 모바일 단일 플랫폼에 집중한다.

```
nexvoy-app (React Native / Expo)
├── 20개 화면 (네이티브)
├── 직접 Supabase SDK 호출
└── 오프라인 캐싱, 네이티브 알림, 스토어 배포
         ↕
   Supabase (공유 DB, 기존 그대로)
```

**이 선택이 맞는 상황:**
- 현재 사용자가 90% 이상 앱을 통해 접근
- 웹에서 의미 있는 유입(SEO, 공유 링크 클릭)이 없음
- 1인 개발에서 두 플랫폼 동시 유지가 품질 저하를 유발하고 있음

**장점:**
- 단일 코드베이스 유지 (RN만)
- 모바일 성능 최대화
- 유지보수 대상이 하나로 집중
- 웹 빌드 복잡도(API Routes 이동 스크립트 등) 완전 제거
- Option C 대비 절반의 공수로 RN 전환 가능

**단점:**

| 항목 | 내용 |
|------|------|
| 웹 접근성 완전 상실 | 브라우저로 접근하는 사용자 완전 차단 |
| 공유 링크 제약 | 현재 `/share/detail` 페이지가 앱 없이 미리보기 불가 |
| 앱 심사 의존 | 긴급 수정도 스토어 심사 대기 필요 (OTA로 완화 가능) |
| 초대/협업 UX 제약 | 앱 미설치 사용자에게 초대 링크 공유 시 진입 장벽 |
| 신규 사용자 유입 제한 | 스토어 검색 + 직접 링크만 허용, 검색 엔진 유입 차단 |

**완화 가능한 단점:**
- 공유/초대 링크: Universal Link + App Store 리다이렉트로 처리 가능
- OTA 업데이트(Expo Updates)로 긴급 배포 가능

**추정 공수:** 2~2.5개월 (Option C 대비 웹 구현 생략)

---

### Option E: API 서버 분리 (Spring/FastAPI) + 웹(Next.js) + 앱(React Native)

웹과 앱 분리에 더해 API 서버까지 별도 운영하는 3-tier 구조.

**OnVoy 현황에서 이 구조의 문제:**
- Supabase가 이미 RLS + Realtime + Edge Functions로 충분한 백엔드 역할 수행 중
- API 서버 추가 시 Supabase의 장점(RLS, 인증, 실시간)을 우회하는 구조가 됨
- 운영 비용 대폭 증가 (서버 호스팅 + 관리)
- 1인 개발 환경에서 세 개 프로젝트 동시 관리는 현실적으로 불가

**결론: 현 시점에서 권장하지 않음.**

## 4. 추천 순위

### 4.1 정식 오픈 전 현재 시점 기준

서비스가 아직 오픈되지 않았다는 사실은 추천 순위를 크게 바꾼다. **운영 사용자가 없으므로 마이그레이션 부담이 없고, 지금이 방향 전환의 최적 타이밍이다.** 잘못된 아키텍처 위에 코드가 더 쌓이기 전에 결정할 수 있다.

| 순위 | 선택지 | 추천 이유 |
|------|--------|---------|
| **1위** | **Option D** (앱 전용 RN) | 오픈 전이라 전환 부담 없음. 모바일 앱으로 집중 출시 후 웹 필요성이 증명되면 추가하는 전략이 리스크 최소 |
| **2위** | **Option C** (웹+앱 분리) | 처음부터 제대로 분리. 웹 유입 채널(SEO, 공유 링크)이 중요하다면 올바른 선택. 단, 1인 개발 지속 부담 존재 |
| **3위** | **Option A** (현상 유지) | 사용자가 없으니 딜레이 문제도 없음. 빠른 출시 후 실제 사용자 데이터를 보고 방향을 결정하는 전략 |
| **비권장** | **Option B** (SSR 강화) | 오픈 전 시점에 복잡도만 늘리는 중간 단계 — 지금은 의미 없음 |
| **비권장** | **Option E** (API 서버 분리) | Supabase로 이미 충분, 비용과 복잡도만 증가 |

### 4.2 추천 판단 기준

**Option D가 1위인 이유:**

오픈 전이라는 조건이 결정적이다. 운영 중인 사용자가 없으므로 전환 시 마이그레이션 부담이 없다. 1인 개발에서 두 플랫폼을 동시에 고품질로 유지하는 것은 장기적으로 더 큰 리스크다. 모바일 앱으로 출시하고, 웹 유입의 필요성이 실제 데이터로 증명된 뒤에 웹을 추가하는 것이 투자 대비 효과가 높다. 공유 링크와 초대 기능은 Universal Link로 대부분 해결된다.

**Option C가 2위인 이유:**

출시 채널로 SEO나 웹 공유 링크를 중요하게 고려한다면 처음부터 분리하는 것이 맞다. 오픈 전 전환이라 마이그레이션 부담도 없다. 단, 두 프로젝트를 동시에 유지하는 비용이 장기적으로 지속된다는 점은 감수해야 한다.

**Option A가 3위인 이유:**

방향을 결정하기 전에 빠르게 출시하고 실제 사용자의 행동을 보는 전략이다. 현재 코드베이스가 이미 동작하고 있으므로 추가 공수 없이 출시 가능하다. 단, 기술 부채 위에서 계속 개발하게 되므로 출시 후 데이터를 보는 시간이 길어질수록 나중에 전환 비용이 늘어난다.

### 4.3 사용자 획득 채널별 판단

어떤 채널로 사용자를 획득할 계획인가가 선택지를 가른다.

| 획득 채널 계획 | 적합한 선택지 |
|--------------|------------|
| 앱 스토어 검색 위주 | Option D (앱 전용) |
| 지인 공유 / SNS 링크 | Option D + Universal Link로 처리 가능 |
| SEO / 블로그 / 콘텐츠 마케팅 | 웹 필수 → Option C |
| 아직 미정 | Option A로 출시 후 채널 확정 시 결정 |

### 4.4 오픈 이후 시점 기준 (참고)

출시 후 사용자 데이터가 쌓이면 아래 기준으로 재평가한다.

| 순위 | 선택지 | 조건 |
|------|--------|------|
| **1위** | **Option A** (현상 유지 + React Query 캐시) | 웹/앱 사용자 분포 불명확한 경우 |
| **2위** | **Option D** (앱 전용 RN) | 앱 사용자 비율 80% 이상 |
| **3위** | **Option C** (웹+앱 분리) | 웹 유입이 유의미하게 성장한 경우 |
| **4위** | **Option B** (SSR 제한적 적용) | 웹 성능 개선만 필요하고 앱은 현행 유지로 충분한 경우 |

## 5. 의사결정 트리

```
정식 오픈 전인가?
  ├── YES (현재 상황)
  │     └── 주요 사용자 획득 채널은?
  │           ├── 앱 스토어 / 모바일 중심 → Option D
  │           ├── SEO / 웹 공유 링크 중요 → Option C
  │           └── 아직 미정              → Option A로 출시 후 결정
  │
  └── NO (오픈 후)
        └── 웹 사용자 비율이 유의미한가? (>20%)
              ├── YES → Option A (캐시 개선) 후 관찰
              │          └── 앱 성능 이슈 지속 시 → Option C
              └── NO  → Option D 검토
                          └── 공유/초대 기능 중요?
                                ├── YES → Universal Link + Option D
                                └── NO  → 완전한 Option D
```

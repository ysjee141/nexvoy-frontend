# ADR-001: 일정 알람 기능 개선 및 서버측 스케줄링 도입 (오프라인 대응 포함)

## 상태
- **상태**: 제안됨 (Proposed)
- **날짜**: 2026-03-25

## 문제 정의
현재 여행 일정(`plans`)별로 시작 전 알람(`alarm_minutes_before`)을 설정할 수 있는 데이터베이스 구조를 갖추고 있으나, 실제 다음과 같은 문제로 인해 알람이 동작하지 않고 있습니다.

1.  **시간 기반 트리거 부재**: 현재는 일정 생성 시에만 트리거가 발생할 뿐, 특정 시간이 되었을 때 알람을 발생시키는 서버측 백그라운드 작업(Cron Job)이 없습니다.
2.  **알림 채널 제한**: 사용자 이메일 알림 기능이 미구현 상태이며, 푸시 알림도 특정 시간에 맞춰 발송되지 않습니다.
3.  **오프라인 상황 대응 부재**: 비행기 모드나 네트워크 불안정 상황(여행 중 빈번함)에서는 서버측 푸시 알림만으로 알람을 보장할 수 없습니다.

## 배경 상황 분석
- **데이터베이스**: `public.plans` 테이블에 `start_datetime_local`, `timezone_string`, `alarm_minutes_before` 컬럼 존재.
- **인프라**:
    - Supabase Edge Function (`send-push-notification`)으로 FCM 푸시 발송 가능.
    - 모바일 앱에 `@capacitor/local-notifications` 및 `@capacitor/push-notifications` 플러그인 설치됨.
- **누락된 요소**: 서버측 스케줄러(Cron) 및 클라이언트측 로컬 알림 동기화 로직.

## 개선 방향 (의사결정)

### 1. 하이브리드 알림 전략 (Hybrid Strategy) 도입
서버측의 광범위한 알림과 클라이언트측의 오프라인 신뢰성을 결합한 하이브리드 방식을 채택합니다.

#### A. 서버측 스케줄링 (Online / Multi-channel)
- **방식**: Supabase Cron을 사용하여 1분마다 `check-pending-alarms` Edge Function을 실행.
- **역할**: 이메일 발송, 멀티 디바이스 푸시 알림, 협업 멤버 대상 알림 담당.
- **장점**: 앱 종료 상태나 다른 기기에서도 알림 수신 가능.

#### B. 클라이언트측 로컬 알림 (Offline Fallback)
- **방식**: 앱이 온라인 상태에서 일정 데이터를 동기화할 때, `@capacitor/local-notifications`를 사용하여 기기 OS 스케줄러에 알람을 미리 등록.
- **역할**: 네트워크가 단절된 상황(비행기 모드 등)에서 실제 기기의 알람 시계 역할 수행.
- **장점**: 네트워크 연결 여부와 상관없이 100% 알림 보장.

### 2. 알람 동기화 및 상태 관리
- **동기화 시점**: 앱 실행 시, 일정 리스트 조회 시, 일정 생성/수정/삭제 시 실시간으로 로컬 알림 큐를 최신화.
- **중복 방지**: 
    - 서버: `plans` 테이블의 `alarm_sent_at` 컬럼으로 중복 발송 방지.
    - 클라이언트: 기기 내 고유 ID(Plan ID 기반)를 사용하여 기존 알림을 덮어쓰거나 취소.

### 3. 알람 중복 및 충돌 해결 (Conflict Resolution)
푸시(Push)와 로컬(Local) 알림이 동시에 발생할 경우 발송될 수 있는 중복 알림 문제를 다음과 같이 해결합니다.

#### A. 고유 ID 기반 그룹화 (Consistent ID)
- **방식**: 일정 ID(`plan_id`)와 알람 유형을 조합한 고유한 정수형 ID를 생성하여 푸시와 로컬 알림에 동일하게 적용합니다.
- **효과**: 안드로이드 등 일부 OS에서는 동일 ID의 알림이 도착하면 기존 알림을 덮어쓰거나 하나만 표시하여 사용자 불편을 최소화합니다.

#### B. 데이터 중심 푸시 (Silent Push) 활용
- **방식**: 서버에서 푸시를 보낼 때 알림(Notification) 페이로드 대신 데이터(Data) 페이로드만 포함된 'Silent Push' 위주로 발송합니다.
- **앱 핸들러 로직**:
    1. 앱이 푸시 데이터를 수신하면, 기기 내 로컬 알람 기록을 확인합니다.
    2. 이미 로컬 알람이 울렸거나 표시 중이라면 푸시 알림을 무시합니다.
    3. 로컬 알람이 아직 울리지 않았고 네트워크가 연결된 상태라면 푸시 요청을 기반으로 알림을 표시합니다.

#### C. 우선순위 설정
- **기기 로컬 알림**: '오프라인 최우선 순위'로 설정하여 네트워크 상태와 무관하게 정확한 시간에 실행되도록 합니다.
- **서버 푸시**: '상호 보완재'로 취급하여 멀티 디바이스 동기화 및 협업 멤버 알림에 집중합니다.

## 기대 효과
- **신뢰성 극대화**: 비행기 모드 등 여행 중 극한의 네트워크 환경에서도 일정이 누락되지 않음.
- **중복 방지**: 하이브리드 전략의 부작용인 '알람 중복 발생'을 기술적으로 억제하여 깔끔한 UX 제공.

## 위험 요소 및 고려사항
- **배터리 최적화**: Silent Push는 OS의 배터리 정책에 따라 지연되거나 무시될 수 있으므로, 로컬 알림의 신뢰성이 절대적으로 중요함.
- **타임존 처리**: 서버(UTC)와 로컬 기기 시간 간의 오차를 방지하기 위해 `timezone_string` 기반의 정확한 시간 계산 로직 필요.
- **로컬 알림 제한**: iOS/Android OS에서 허용하는 최대 로컬 알림 개수(일반적으로 64~100개) 내에서 효율적으로 큐를 관리해야 함.

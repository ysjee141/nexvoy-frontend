# 🗄️ 캐싱 정책 (Caching Policy)

이 문서는 NexVoy 애플리케이션의 캐싱 목적과 운용 정책을 정의합니다. 과거 혼재되어 있던 캐싱 정책을 전면 재수립하여 성능 향상과 오프라인 지원 목적을 명확히 분리합니다.

## 1. 개요 및 문제 인식

### 1.1. 기존 캐싱의 문제점
1. **계정 간 캐시 오염**: 다중 계정 로그인/로그아웃 전환 시 이전 사용자의 캐시된 데이터가 일시적으로 노출됨.
2. **오프라인 지원 미흡**: 오프라인 상태에서 기존 로컬 스토리지 캐시 데이터가 정상적으로 표시되지 않거나 일관성이 없음.
3. **생명주기 불분명**: 캐시의 생성, 갱신, 무효화(삭제) 시점에 대한 명확한 기준 부재.
4. **목적의 혼재**: 단순 API 성능 향상을 위한 캐시인지, 네트워크 불안정/오프라인 환경을 대비한 캐시인지 구분되지 않음.

---

## 2. 목적별 캐싱 분리 정책

캐싱의 목적을 **'성능 향상'**과 **'오프라인/네트워크 불안정 대비'** 두 가지로 명확히 분리하여 각각 다른 저장소와 생명주기를 가지도록 설계합니다.

### 2.1. 성능 향상을 위한 캐싱 (Performance Caching)
API 응답 속도를 개선하고 불필요한 네트워크 요청을 줄이기 위한 캐싱입니다.

* **저장 위치**: Server-Side 캐싱 활용 (현재 Local Storage 사용 배제)
* **Vercel 환경 고려사항**:
  * Vercel 배포 환경의 다중 인스턴스 간 공유 이슈를 해결하기 위해 **Next.js 내장 캐시(Data Cache)**를 적극 활용.
  * Next.js Data Cache는 Vercel의 Data Cache 인프라를 통해 배포 환경에서 인스턴스 간 글로벌 공유 및 지속성(Persistence)을 보장함.
* **적용 대상**: 변경 빈도가 낮고 여러 사용자가 공통으로 조회할 수 있는 메타 데이터 등.
* **생명주기**:
  * 계정 전환 시 클라이언트 측에서 들고 있는 SWR/React Query 등 메모리 캐시는 철저히 초기화(`mutate`, `clear`) 필수.

### 2.2. 오프라인 및 네트워크 불안정을 위한 캐싱 (Offline Caching)
사용자가 네트워크에 연결할 수 없는 상황에서도 본인의 여행 계획을 열람할 수 있도록 지원하기 위한 캐싱입니다.

* **저장 위치**: Client-Side (Local Storage / IndexedDB 등)
* **적용 대상**: 넷플릭스의 콘텐츠 다운로드 기능과 유사하게, **사용자가 명시적으로 다운로드를 요청한 "여행 단위(Trip)"의 데이터** (여행 정보, 상세 일정, 체크리스트 등). 단순 열람만으로는 자동 캐싱되지 않음.
* **다운로드 관리 (프로필 페이지)**:
  * 프로필 페이지 내에 사용자가 다운로드한 오프라인 여행 목록을 확인하고, 기기 용량 확보를 위해 삭제(정리)할 수 있는 관리 기능 제공.
* **앱 전용 기능 (App-Only Policy)**:
  * 오프라인 모드는 브라우저(Web) 환경에서는 지원하지 않으며, **오직 Capacitor 기반 모바일 앱 환경에서만 제공**함.
* **오프라인 전용 라우트 분리 (`/offline/...`)**:
  * 온라인 페이지(예: `/trips/[id]`)를 오프라인 상태에서 폴백시키는 것이 아니라, 오프라인 전용 페이지(`/offline/trips/[id]`)를 신설하여 완전히 분리함.
  * 신설된 오프라인 라우트들은 클라이언트의 전역 인증(Auth) 가드를 우회(Bypass)할 수 있도록 예외 처리함.
  * **UI 컴포넌트 리팩토링**: 이 구조를 위해 데이터 통신(Data Fetching) 로직을 분리하여 순수 UI 컴포넌트(Presentational Component)를 만들고, 온라인 페이지와 오프라인 전용 페이지가 동일한 UI 컴포넌트를 `props`를 주입하여 재활용하도록 리팩토링함.
* **동기화 및 렌더링 전략 (Network-First & 명시적 전환)**:
  * **온라인 상태**: 네트워크 요청을 우선 수행(로딩 UI). 성공 시 화면 렌더링 후 백그라운드에서 로컬 캐시를 조용히 갱신.
  * **오프라인 모드 진입 (사용자 선택)**:
    1. 앱 동작 중 전역 네트워크 감지기(Capacitor Network Plugin)를 통해 단절/불안정 감지.
    2. 자동 폴백(Fallback) 대신 "네트워크가 불안정합니다. 다운로드한 오프라인 여행 모드로 전환하시겠습니까?"를 묻는 프롬프트(모달/바텀시트) 제공.
    3. 사용자가 동의 시에만 `/offline` 라우트로 명시적 전환.
* **오프라인 모드 정책 (Read-Only)**:
  1. **읽기 전용 모드**: 여행, 일정, 체크리스트 등 캐시된 모든 데이터는 읽기 전용으로만 노출.
  2. **수정 기능 차단**: 임시 저장 후 온라인 전환 시 동기화(Sync)하는 복잡도를 피하기 위해, 데이터의 추가/수정/삭제와 관련된 모든 UI(버튼, 입력 폼 등)는 비활성화(Disabled).
* **생명주기 및 격리**:
  * 로컬 스토리지에 저장 시 반드시 사용자의 고유 식별자(`user_id`)를 기준으로 분리하여 저장.
  * 로그아웃 동작 발생 시 해당 기기에 남아있는 오프라인 캐시 데이터를 즉각 파기하여 계정 간 데이터 유출을 방지.

---

## 3. 요약 및 향후 과제

본 정책은 데이터 무결성을 최우선으로 고려합니다. 
성능 목적의 데이터는 서버 측에서 관리하고, 오프라인용 데이터는 로컬에 유지하되 철저한 '읽기 전용' 모드로 제공하여 복잡도를 낮추고 사용자 경험을 보호합니다.

- **Action Item**: Next.js 내장 캐시(Data Cache)를 활용한 Server-side 성능 캐싱 구현.
- **Action Item**: 사용자의 명시적인 '여행 다운로드' 액션 흐름 및 프로필 페이지 내 다운로드 관리 UI 구현.
- **Action Item**: 앱 내 전역적인 네트워크 상태(Online/Offline) 감지 로직 구현 및 모드 선택 UI 추가.
- **Action Item**: 사용자 로그아웃 흐름에서 Client-Side 스토리지 전면 초기화 로직 점검 및 기존 무분별한 로컬 캐싱 걷어내기.

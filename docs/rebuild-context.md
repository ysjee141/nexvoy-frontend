## 서비스의 재개발 의사결정을 위한 분석

### 배경

현재 이 프로젝트는 NextJs + Supabase Backend(BaaS), 앱 개발을 위해 capacitor을 사용하고 있어. 물론 DB 또한 Supabase를 사용하고 있어.
현재의 이 구성으로 한 벌의 개발을 통해 웹 / 앱을 모두 지원하고 있기 때문에 나쁘지 않은 선택이라고 생각해.

다만, 사용성과 반응성 측면에서 이게 최선인가에 대한 의문이 들어. 이렇게 생각하는 이유는 다음과 같아.

* 여행 일정 페이지를 들어가게 되면 SSG이기 때문에 데이터를 읽고 표시하는데까지 시간이 걸리고, 그것이 사용자의 눈에 확인이 가능해 (.5s~1s)
* 이건 사용자의 경험상 그리 좋지 않은 선택일 것이라고 생각해
* 앱에 특화되어 있는 기능 (오프라인 모드, 네트워크 상태 감지 등)이 따로 존재하고 있어

### 의사결정이 필요한 점

사용성과 반응성 측면에서 최적의 결과를 얻기 위해서는 결국 두벌로 관리가 되어야 하지 않을까 싶어

1. NextJs SSR을 통한 웹 개발
2. React Native를 통한 앱 개발 (Android, iOS 모두 지원을 위해)

### 필요할 것

* 이 프로젝트 전체를 분석해서 앞서 이야기한 방식의 유효성을 증명
* 수정의 범위의 확정이 필요. React Native를 통한 앱개발이라면 API가 별도로 분리되어야 하는 상황도 고려
  * 그렇다고 한다면, API 서버를 따로 두고 (Spring이든 뭐든), 웹은 NextJS SSR / 앱은 React Native로 개발하는 것도 고려

### 현 운영환경

* vercel - NextJs 배포 (Free plan)
* supabase - Backend(BaaS) (Free plan)

### Rebuilding 이후 환경

* AWS / GCP / Vercel / Supabase 어떠한 것이든 상관없음
* 최소 비용으로 운영할 수 있는 방법이 필요함
